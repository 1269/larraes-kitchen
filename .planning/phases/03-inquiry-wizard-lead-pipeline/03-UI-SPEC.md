# Phase 3 — UI Design Contract

> Visual and interaction contract for the Inquiry Wizard & Lead Pipeline phase. All decisions below are **locked** for executor consumption — spacing, typography, color usage, copywriting, stepper, focus management, validation affordances, and mobile behavior. Generated by gsd-ui-researcher; verified by gsd-ui-checker.

**Scope:** 4-step wizard inside a full-screen shadcn Dialog (Event Type → Guests & Date → Package → Contact), live price-range sticky bar, confirmation view with submission ID, all field/step/server error states, Turnstile widget placement, honeypot + min-time bot defenses, idempotent submit handling.

## Spacing Scale

Declared values (multiples of 4; source: existing Phase 2 component conventions — HeroSection, PackagesSection, ContactSection use `px-5 md:px-8`, `gap-6 md:gap-8`, `mt-12 md:mt-16`, `py-16 md:py-24 lg:py-32`):

| Token | Value | Wizard Usage |
|-------|-------|--------------|
| xs | 4px | Icon-to-label gap in chip, star rating spacing, inline form helper padding |
| sm | 8px | Gap between stepper dots, form field internal padding, error icon gap |
| md | 16px | Default gap between form field rows, inside tile padding, card internal spacing, sticky bar internal padding |
| lg | 24px | Gap between major wizard sub-sections (step heading → field group → actions), Dialog content internal padding (mobile), card padding mobile |
| xl | 32px | Dialog content padding (desktop), spacing between Dialog header and step content |
| 2xl | 48px | Reserved — not used in wizard (modal is always dense by design) |
| 3xl | 64px | Reserved — not used in wizard |

**Exceptions (locked):**
- **Touch target floor: 44×44px (WIZ-08).** Apply `min-h-[44px] min-w-[44px]` to every tappable element in the wizard: chips, persona tiles, stepper dots, Next/Back buttons, radio rows, submit. Button padding increases over the shadcn `h-9` default where needed to clear 44px; `size="lg"` with `min-h-[44px]` wrapper class is the pattern.
- **Sticky estimate bar height: 72px on mobile, 80px on desktop** (allows two lines of body-md text plus 16px vertical padding).
- **Dialog padding at ≤640px viewport: 16px** (not 24) — edge-to-edge feel is prioritized over generous whitespace on small phones (Larrae's users are mobile-first per persona doc).
- **Package tier card internal padding in Step 3: 24px mobile, 32px desktop** — matches existing `PackagesSection.astro` `p-6 md:p-8` pattern so the wizard card feels like the static packages card the user just came from.

## Typography

**Wizard type scale: exactly 4 distinct pixel sizes — 14, 16, 18, 32.** The 12px `text-eyebrow` token remains available for page-level section eyebrows outside the wizard, but is deliberately excluded from the wizard surface to keep the modal's type hierarchy tight.

| Role | Token | Size | Weight | Line Height | Wizard Usage |
|------|-------|------|--------|-------------|--------------|
| Step heading (H2 per step) | `font-display text-display-md` | 32px (2rem) | 400 (Lovelace is a single weight) | 1.15 | "Tell us about your event", "How many, and when?", "Pick a package", "How should we reach you?", and "Thanks, {name} — your request is in." on confirmation |
| Step subtitle / micro-label (eyebrow) | `text-body-sm uppercase tracking-[0.12em] font-semibold` | 14px (0.875rem) | 600 | 1.5 | "STEP 1 OF 4", "STEP 2 OF 4", … above each step heading. Uppercase + tight tracking + semibold carry the eyebrow signal; font size is **not** doing that work here so the wizard keeps a 4-size scale. Color: `text-accent` (deep amber) to match the site-wide eyebrow language. |
| Body (default) | `text-body-md` | 16px (1rem) | 400 | 1.5 | Field labels, helper text, tile descriptors, chip labels, sticky-bar primary line, confirmation recap rows. **16px floor prevents iOS zoom on input focus (WIZ-07 compliance).** |
| Body large (emphasis) | `text-body-lg` | 18px (1.125rem) | 400 | 1.5 | Sticky-bar estimate range (visual anchor line), confirmation submission ID `LK-XXXXXX` display |
| Body small (helper / meta / eyebrow) | `text-body-sm` | 14px (0.875rem) | 400 (600 when used as eyebrow) | 1.5 | Inline validation error text, lead-time + blackout-date hint below date picker, "required" markers, "recommended for N guests" sub-caption on Step 3 cards, "Leave the quote wizard?" confirmation body. **Weights declared: 2 total.** Work Sans at **400 (regular)** for body, and **600 (semibold)** for field labels, chip active state, stepper active state, eyebrow labels, and CTA text. Lovelace (display) ships at a single weight — not counted as a separate weight decision. Playfair Display used italic for prices only.

**Heading line-height: 1.15** (CSS: `--text-display-md` inherits the declared `1.05` for `display-xl`; step headings use `leading-tight` utility = 1.25 explicitly to avoid cramped ascenders of Lovelace at 32px — executor applies `leading-tight` on step `<h2>`).

**Body line-height: 1.5** (Tailwind default for body sizes — no override).

## Copywriting Contract

**All copy below is LOCKED.** Executor writes these strings verbatim. Parameterized inserts are in `{braces}`.

### Primary CTAs (per step)

| Element | Copy |
|---------|------|
| Primary CTA (Step 1) | `Next — add guests & date` |
| Primary CTA (Step 2) | `Next — pick your package` |
| Primary CTA (Step 3) | `Next — your contact info` |
| Primary CTA (Step 4, idle) | `Send my request` |
| Primary CTA (Step 4, loading) | `Sending…` (button disabled, spinner icon) |
| Secondary action (Steps 2–4) | `Back` |
| Dialog close (top-right icon button) | `Close` (visually an X; `aria-label="Close quote wizard"`) |
| Confirmation view primary action | `Back to site` |
| Estimate bar secondary CTA (none — display-only) | — |

**"Back" label rationale (Dimension 1 non-blocking flag):** The single-word `Back` on Steps 2–4 is an intentional departure from the noun-verb pattern used on primary CTAs. Rationale:
1. In a linear modal wizard with a visible progress stepper, the directional signal is already carried by the stepper — adding "Back to guests" / "Back to package" is redundant with the numbered step breadcrumbs overhead.
2. Consistency across steps: a single stable secondary-action label reduces cognitive load on return visits.
3. The primary-action (green, full-width on mobile) already carries the noun'd hand-off ("Next — pick your package"), so the secondary slot can stay compact and the CTA stays the visual anchor.
4. `Back` universally means "previous step" in the Radix Dialog focus trap context (WCAG understandable).

### Live estimate copy (D-10, D-13)

| State | Copy pattern |
|-------|-------------|
| Insufficient data (before guest count entered) | **Bar hidden entirely.** No "$0–$0" flash. |
| Valid estimate (10–75 guests) | Line 1 (large): `Estimated ${min}–${max}` — e.g. `Estimated $500–$650`. Line 2 (smaller, same visual weight per EST-04): `Final quote confirmed by Larrae`. |
| Out-of-range / custom-quote (D-12) | `Custom quote — Larrae will follow up` (single line centered) |
| Loading (debounce pending on large paste) | Line content unchanged but opacity 70% + tiny `Calculating…` aria-live text |

**ARIA live-region semantics (A11Y-03, A11Y-05, EST-07):**
- Sticky estimate bar wraps its number line in `aria-live="polite"` `aria-atomic="true"` + `role="status"`.
- Debounce on guest input is **250ms** (per D-13 / EST-07) — screen readers announce the settled value, not every keystroke.
- On Step 3 tier toggle, estimate updates immediately (no debounce on tier — user-initiated discrete change).
- When transitioning to "Custom quote — Larrae will follow up" state, announce once; do not thrash.
- Field-level validation errors render inside `<p id="{field}-error" role="alert">` wired via `aria-describedby={field}-error` + `aria-invalid="true"` on the input.

### Submission failure / server error alerts (D-18)

Alert block appears **above** the Step 4 submit button when a server error is returned. Pattern: `role="alert"`, red-tinted background, icon (`AlertTriangle`), inline `mailto:` link built from `site.email`. Stays visible until user edits a field OR resubmits.

| Error type | Alert copy |
|-----------|-----------|
| Turnstile verification failed (LEAD-02) | `Having trouble verifying you're human. Please try again, or email us directly at [{site.email}](mailto:{site.email}).` |
| Rate limit hit (LEAD-03) | `Too many attempts — please wait a few minutes, or email us directly at [{site.email}](mailto:{site.email}).` |
| Idempotency replay (LEAD-04) — handled silently | No alert — UI skips to confirmation view with the prior submission ID. |
| Server 5xx | `Something went wrong on our end — please try again, or email us directly at [{site.email}](mailto:{site.email}).` |
| Network timeout | `We couldn't reach our servers. Check your connection and try again, or email us at [{site.email}](mailto:{site.email}).` |

### Silent bot-reject (SPAM-01, SPAM-03, SPAM-04) — D-18

**Indistinguishable-from-real-success UI.** Server returns a 200 with a decoy `LK-XXXXXX` ID that is never persisted; the wizard flips to the confirmation view with that decoy ID. No tell: same copy, same layout, same screen. No emails sent. No lead record written. The bot cannot distinguish acceptance from silent rejection.

### Empty state

## Interaction & Motion

**All motion respects `prefers-reduced-motion: reduce` (WIZ-14).** When reduced motion is set, every transition and animation collapses to instant state swap. Implementation note for executor: use the project's `motion-safe:` Tailwind variant or a `@media (prefers-reduced-motion: reduce) { * { animation: none; transition: none; } }` scope inside the wizard root.

| Interaction | Default motion | Reduced-motion behavior |
|------------|----------------|------------------------|
| Dialog open | Fade backdrop 200ms + scale content 95→100% 200ms (Radix Dialog defaults) | Instant |
| Step transition (Next / Back) | Fade-slide 150ms (content cross-fade, step number updates simultaneously) | Instant (just re-render) |
| Sticky bar appearance | Fade + slide up 200ms when first visible valid estimate appears | Instant (set `opacity-100`) |
| Estimate number update | No animation on number change (avoid attention-hijacking pulse — estimate fires frequently). Optional: 100ms opacity dip for visual acknowledgment only. | Instant |
| Stepper dot activation | 150ms background color transition | Instant |
| Chip selection | 100ms background transition | Instant |
| Validation error appear | Slide-down 120ms from above field | Instant |
| Confirmation view reveal (post-submit success) | Cross-fade 250ms from form to confirmation | Instant |
| Submit button loading state | Spinner icon rotates at 1s/rotation via `animate-spin` | Reduced-motion: static spinner icon (still shows loading, just not rotating) |

**Focus management (A11Y-03, WIZ-13) — locked order:**

1. On Dialog open: focus lands on the Dialog heading (`<h2 id="wizard-title">Get a quote</h2>`, visually hidden to screen readers but focusable via `tabindex="-1"`). shadcn Dialog's built-in `autoFocus` on first focusable element would land on the X button — we override to the heading for clearer screen-reader context.
2. On step advance (Next click, validation passes): focus moves to the new step's `<h2>` (same pattern — `tabindex="-1"` on headings for programmatic focus).
3. On step blocked (Next click, validation fails): focus moves to the **first invalid field** and the error-summary live region at the top of the step announces the error count.
4. On step back (Back click): focus moves to the new step's `<h2>`.
5. On Turnstile widget interaction: focus stays on widget; Tab advances to submit button.
6. On submit success: focus moves to confirmation view `<h1>`.
7. On submit error: focus moves to the inline error alert (`role="alert"` auto-announces; `tabindex="-1"` + programmatic focus for sighted keyboard users to find it).
8. On Dialog close (via X, Escape, or Back-to-site): focus returns to the element that opened the Dialog (Radix Dialog default — verify in tests).

**Keyboard shortcuts:**
- `Tab` / `Shift+Tab`: natural DOM order within the current step (Radix Dialog focus trap)
- `Escape`: triggers dirty-dismiss flow if dirty, else closes Dialog
- `Enter` inside a field: submits the Next action (or Submit on Step 4) — natural form behavior
- `Arrow keys` on persona tiles (Step 1) and tier cards (Step 3): horizontal navigation between tiles/cards (ARIA radiogroup pattern)
- `Space` on chip buttons (Step 2): toggles chip selection

## Responsive Behavior

**Mobile-first (LAYT-03). Breakpoints inherited from Tailwind defaults: `sm` 640, `md` 768, `lg` 1024.**

| Element | Mobile (<640px) | Tablet (640–1024px) | Desktop (≥1024px) |
|---------|-----------------|---------------------|-------------------|
| Dialog content | Full viewport, edge-to-edge, padding 16px. `h-dvh w-dvw max-w-none` (dynamic viewport height for mobile Safari) | Centered modal, 640px wide, padding 24px, rounded-lg, max-h 90vh with scroll | Centered modal, 720px wide, padding 32px, rounded-lg |
| Progress stepper | Top of dialog, horizontal, dots only (compact), eyebrow label "STEP 2 OF 4" as separate text below | Same | Same |
| Persona tiles (Step 1) | Stacked vertical, full width, 96px tall each | 3 tiles side-by-side, equal width | Same as tablet |
| Guest chips (Step 2) | 2×2 grid, full-width chips (each 44px tall minimum) | Horizontal row, wrap if needed | Horizontal row |
| Date picker | Full-width native input (triggers device date UI on iOS/Android) | Same | Same |
| Package cards (Step 3) | Stacked vertical, full-width; collapsed details (show name + price + 2-line inclusion summary + expand toggle) | 3 cards side-by-side, always-expanded | Same as tablet |
| Contact fields (Step 4) | Single column, full width | Two-column for related pairs (Name+Phone, Email+Preferred-contact) | Same |
| Turnstile widget | Centered, 300px max width | Same | Same |
| Sticky estimate bar | Fixed to bottom of Dialog content (above the form's Next/Back buttons) with `sticky bottom-0`, full width of Dialog, white background with top border + subtle upward shadow | Same (sticky inside the contained modal) | Same |
| Submit + Turnstile grouping | Turnstile above submit; legal/privacy one-liner below submit; all three in same visual block | Same | Same |
| Bottom action row (Back / Next) | Full-width stacked: Next on top (primary), Back below (text-style) | Side-by-side: Back left, Next right | Same |

**iOS-specific:** 16px minimum font-size on all inputs (enforced by `text-body-md` = 1rem). Never use `font-size: 14px` on an input or iOS will zoom the viewport on focus (PERF regression + UX breakage).

**Android-specific:** Native date picker + native select. No custom picker JS. No dropdown polyfill.

**Gesture + keyboard parity:** Every tile, chip, and card is both tap- and keyboard-activated. Never use `onMouseOver`-only affordances. Backdrop click to close the dialog is equivalent to pressing Escape.

## Analytics Integration (OBS-01 hook)

The following analytics events are fired client-side by the wizard. Event naming is locked here so the dashboards defined in Phase 4 can reference stable identifiers.

| Event | Trigger | Properties |
|-------|---------|-----------|
| `wizard_start` | Dialog first opens (per session) | `entryPoint`: `'hero'` / `'nav'` / `'package_card'` / `'contact_section'`; `tierPreselected?`: `'small'` / `'medium'` / `'large'` |
| `wizard_step_complete` | Successful Next click | `step`: `1` / `2` / `3` / `4`; `eventType?`, `guestCount?`, `packageId?` as they become known |
| `wizard_submit_success` | Server returns submission ID | `packageId`, `guestCount`, `eventType`, `estimateMin`, `estimateMax` (for conversion attribution) |
| `wizard_submit_failure` | Server returns error (non-bot) | `reason`: `'turnstile'` / `'rate_limit'` / `'server_error'` / `'network'` |
| `wizard_dismiss_dirty` | User confirms dismiss with form dirty | `step`: last-completed step |

**PII discipline:** No `email`, `phone`, `name`, or free-text `notes` are ever passed to analytics. Only the shape of the funnel and conversion metadata.
